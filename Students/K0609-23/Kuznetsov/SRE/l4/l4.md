# Отчет по лабораторной работе: Логирование и Observability

### Инициализация и настройка среды
Выполняется подготовка инфраструктуры с помощью скрипта `setup-logging.sh`. В ходе работы скрипта проверяется наличие необходимых образов и запускается стек из четырех контейнеров: `practic-app-1`, `practic-nginx-1`, `practic-postgres-1` и `practic-rate-limiter-1`. Особое внимание уделяется проверке доступности сервисов по сети: с помощью `curl` подтверждается, что Nginx корректно проксирует запросы на лимитер, а лимитер, в свою очередь, взаимодействует с upstream-приложением.

---

### Организация системы логирования
В рамках работы настраивается двухканальная система сбора данных. Основным инструментом выступает сервис лимитера, написанный на Node.js.

* **Файловое логирование**: журналы событий в формате JSON записываются в директорию `/app/logs`. На скриншотах выполнения зафиксировано использование команды `tail -f` для чтения файла `requests-[дата].log`, что позволяет отслеживать параметры каждого HTTP-запроса: метод, путь, IP клиента и время отклика.
* **База данных**: параллельно данные экспортируются в PostgreSQL. В таблице `request_logs` фиксируются все транзакции, включая те, что были заблокированы алгоритмом ограничения частоты запросов.

---

### Практическое тестирование и анализ трафика
Проводится серия тестов для проверки работоспособности лимитера. С помощью `curl -v` анализируются заголовки ответов. При штатной нагрузке сервер возвращает HTTP 200, а при искусственном создании очереди запросов фиксируется переход лимитера в режим блокировки с выдачей кода 429 Too Many Requests.

На скриншотах видно, что запросы к эндпоинтам типа `/health` и `api/status` обрабатываются успешно.

---

### Аналитическая работа с данными в СУБД
Для извлечения метрик производительности выполняется прямое подключение к базе данных через `docker-compose exec`. Проводится агрегация накопленной информации с использованием SQL-запросов:

* **Статистика по кодам ответов**: выполняется группировка (`GROUP BY status_code`), позволяющая увидеть распределение между успешными запросами, ошибками 502 (Bad Gateway) и блокировками 429.
* **Временные показатели**: рассчитывается среднее (`AVG`) и максимальное (`MAX`) время обработки запроса, что помогает выявить "узкие места" в цепочке проксирования.
* **Проверка лимитов**: отдельными запросами извлекаются записи, где флаг `rate_limited` установлен в значение `true`, для подтверждения корректности работы алгоритма Token Bucket.

### Устранение неисправностей и особенности реализации
В процессе выполнения работы были выявлены и решены следующие проблемы:

* **Ошибка «input device is not a TTY»**: при попытке выполнить SQL-запрос через перенаправление ввода (`<< EOF`) в Docker-контейнер возникала системная ошибка терминала. Проблема решена добавлением флага `-T` в команду `docker-compose exec`, что обеспечило стабильную передачу команд в неинтерактивном режиме.
* **Локализация "немых" логов**: обнаружено, что стандартный вывод `docker-compose logs` не содержит подробностей о запросах. Путем анализа файловой структуры и кода `middleware.js` определено, что логи пишутся исключительно в файлы и базу данных, после чего мониторинг был перенастроен на работу с этими источниками.
* **Конфликты доступа в БД**: на основе анализа `docker-compose.yml` и вывода ошибок `psql` была скорректирована строка подключения (уточнены имя пользователя `postgres` и база данных `demo`), что позволило восстановить запись логов после кратковных сбоев подключения.
