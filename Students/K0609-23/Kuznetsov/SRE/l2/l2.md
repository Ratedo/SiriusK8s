### Отчет "Docker (безопасность) и Multi-stage Docker Build"

**1. Проверка текущего состояния безопасности**
С помощью специализированного скрипта `check-docker-security.sh` проводится комплексный анализ конфигурации запущенных контейнеров. В ходе проверки выявляются критические уязвимости (статус **CRITICAL**): процессы в контейнерах `app`, `postgres`, `rate-limiter` и `nginx` запускаются от имени суперпользователя (**root**). Также фиксируется отсутствие ограничений по системным возможностям (capabilities) и использование доступной для записи корневой файловой системы.

**2. Перевод сервисов на запуск от не-привилегированных пользователей**
Для устранения выявленных рисков выполняется корректировка конфигурационных файлов:
* В `Dockerfile.middleware` вносятся инструкции по созданию системного пользователя и группы `node`, назначению прав на рабочую директорию и переключению контекста исполнения через инструкцию `USER`.
* В файле `docker-compose.yml` для сервиса `nginx` устанавливается идентификатор пользователя `101:101`.
* Для сервиса базы данных (`postgres`) настраивается запуск с использованием системного UID `70:70`.
Повторное сканирование подтверждает успешный переход всех сервисов в режим **non-root user**.

**3. Ограничение системных привилегий**
Для минимизации прав доступа контейнеров к ресурсам хоста в конфигурацию `docker-compose.yml` вносятся следующие изменения:
* Применяется параметр `cap_drop: ALL`, который полностью отключает все Linux Capabilities для каждого сервиса.
* В блоке `security_opt` активируется опция `no-new-privileges:true`, исключающая возможность повышения прав процессами внутри контейнера.

**4. Настройка неизменяемой файловой системы**
Реализуется концепция **Read-only Root Filesystem**. Для всех сервисов в манифесте устанавливается параметр `read_only: true`, запрещающий запись в корневой раздел контейнера. Работа сервисов, требующих записи временных данных, обеспечивается через механизм `tmpfs` (монтирование участков оперативной памяти):
* Для `nginx` выделяются области `/var/cache/nginx`, `/var/run` и `/tmp`.
* Для `postgres` настраиваются пути `/tmp` и `/var/run/postgresql`.
Внешние конфигурационные файлы монтируются с флагом `:ro` (read-only) для защиты от модификации.

**5. Оптимизация сборки образа (Multi-stage Build)**
Для сокращения размера итогового образа и удаления инструментов разработки применяется многоэтапная сборка в `Dockerfile.app`:
* **Stage 1 (builder):** На базе образа `golang:1.21-alpine` выполняется компиляция исходного кода в бинарный файл.
* **Stage 2 (runtime):** На базе минимального образа `alpine:latest` формируется финальное окружение. Из предыдущего этапа копируется только исполняемый файл `app`.
Сборка запускается с явным указанием целевого этапа `--target runtime`.

**6. Контроль итоговых показателей**
По завершении настройки проводится финальная проверка. Скрипт подтверждает отсутствие критических замечаний и предупреждений (статус **No critical issues or warnings found!**). Сравнение характеристик образов показывает, что применение Multi-stage сборки позволяет уменьшить размер образа `app` до **34.4 MB**, что значительно ниже.
