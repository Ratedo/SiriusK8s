### Отчет "Тестирование Nginx"

---

**1. Настройка окружения и устранение ошибок доступа к Docker**
При первичной попытке запуска стека сервисов через `docker-compose up -d` фиксируется ошибка доступа к сокету Docker (`permission denied`). Для обеспечения корректной работы без использования прав суперпользователя производится добавление текущего пользователя в группу `docker`. С помощью команды `newgrp docker` обновляется конфигурация групп в текущей сессии, после чего доступ к Docker-демону восстанавливается.

**2. Конфигурация инфраструктуры**
Производится анализ файла `docker-compose.yml`. В состав инфраструктуры входят четыре сервиса:
* **postgres**: база данных с настроенной проверкой состояния (`healthcheck`) и монтированием скриптов инициализации.
* **app**: основное приложение на языке Go, собираемое из локального Dockerfile. Установлена зависимость от состояния готовности базы данных.
* **rate-limiter**: дополнительный модуль для ограничения частоты запросов.
* **nginx**: реверс-прокси для маршрутизации входящего трафика.

**3. Устранение ошибок компиляции приложения**
В процессе сборки образа приложения (`docker compose up -d --build`) возникает ошибка на этапе выполнения `go build`. Компилятор прерывает процесс из-за наличия импортированного, но не используемого пакета `os` в файле `main.go`. 
Для решения проблемы производится редактирование исходного кода: строка импорта пакета `os` комментируется. После внесения правок повторная сборка и запуск контейнеров завершаются успешно.

**4. Запуск и проверка состояния сервисов**
Выполняется развертывание всех компонентов системы. С помощью команды `docker ps` осуществляется мониторинг статуса контейнеров. Наблюдается процесс перехода сервисов в состояние `healthy` после прохождения внутренних проверок. 
Для ручной проверки работоспособности выполняется запрос через `curl` к эндпоинту `http://localhost:8080/health`. Приложение возвращает корректный JSON-ответ со статусом `healthy` и актуальной меткой времени.

**5. Итоговое тестирование**
Для комплексной проверки системы запускается автоматизированный скрипт тестирования `test-api.sh`. Сценарий проверяет доступность узлов, корректность заголовков, обработку неверных запросов и время ответа. По результатам выполнения тестов фиксируется успешное прохождение всех 16 проверок (Passed: 16, Failed: 0). Система признается полностью работоспособна и соответствует требованиям.
